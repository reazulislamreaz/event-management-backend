# Prisma schema review — event wiki & sessions

## সংক্ষেপ (বাংলা)

আগের `Event` মডেলে **একই নামে দুইবার `createdBy` ফিল্ড** ছিল (একবার `String`, আবার `String?`) — এটা Prisma তে **ইনভ্যালিড**। পাশাপাশি `User` মডেলে `events` আর `createdEvents` দুটোর মধ্যে `Event` এর দিকে রিলেশন মিলছিল না, তাই `prisma validate` ফেল করত। এগুলো ঠিক করে `creatorId` / `lastEditorId` দিয়ে wiki-স্টাইল অডিট স্পষ্ট করা হয়েছে; `EditLog` এ এখন `editorId` → `User` ফরেন কি আছে যাতে contribution স্কোরিং পরে সহজ হয়।

---

## Critical issues that were fixed

### 1. Duplicate field `createdBy` on `Event`

The schema defined `createdBy` twice (required `String` and optional `String?`) plus overlapping `user` / `userId` / `createdByUser` relations. Prisma rejects duplicate field names (`P1012`).

**Resolution:** Single ownership model:

- `creatorId` + `creator` — `@relation("EventCreator")`, `onDelete: Restrict` (events keep a valid creator reference; align with soft-delete users in app if needed).
- `lastEditorId` + `lastEditor` — `@relation("EventLastEditor")`, `onDelete: SetNull` (optional last editor if user row is removed).

On create, set `lastEditorId` to the same user as `creatorId` in application code.

### 2. `User` ↔ `Event` relation mismatch

`User` had both `events` and `createdEvents` with `@relation("EventCreator")` on only one side, while `Event.user` used the default unnamed relation. That broke the relation pairing rules.

**Resolution:** `User` now exposes:

- `eventsCreated` → `EventCreator`
- `eventsLastEdited` → `EventLastEditor`

Remove any old references to `user.events` or `user.createdEvents` in queries; use `eventsCreated` / `eventsLastEdited`.

### 3. `EditLog` editor not linked to `User`

`editedBy` was a loose `String`, which makes contribution scoring and joins harder and allows typos/orphan IDs.

**Resolution:** `editorId` + `editor` with `onDelete: Restrict`. Hard-deleting a user who has edit logs will fail until logs are migrated or the user is soft-deleted only — consistent with audit trails.

## Other design notes (not bugs, but watch in production)

| Topic | Risk / recommendation |
|--------|------------------------|
| **`User.contributionScore`** | Updating a single counter from many places is race-prone. Prefer a **`ContributionLedger`** (append-only rows: action, points, `eventId`, `editLogId`) and derive or periodically reconcile the score. |
| **“Verify others”** | Not modeled yet. Add e.g. `EventVerification` or `EditReview` (reviewer, target `EditLog` or `Event` version, status) when you implement that flow. |
| **Auto next session from `RepeatConfig`** | `RepeatConfig` stores frequency and `nextAutoGenerateDate`; **generating** the next `Session` / `EventSession` is application or cron logic, not automatic in the DB. |
| **`Subcategory.name` `@unique`** | Uniqueness is global, not per `categoryId`. Two categories cannot both have a subcategory named `"Junior"`. Often teams use `@@unique([categoryId, name])` instead. |
| **`Program.name` `@unique`** | Global uniqueness for program titles — confirm that matches product rules. |
| **`EventSession.programId`** | Denormalized copy of `Event.programId`; keep in sync in the service layer or document why it may diverge. |
| **`Session` + `EventSession`** | `onDelete: Restrict` on `EventSession.session` prevents deleting a `Session` that is still linked to events. |

## Migration reminder

If you already have a database with the old `Event` / `EditLog` columns, create a migration that:

1. Adds `creatorId`, `lastEditorId`, `editorId` (nullable first if backfilling).
2. Backfills from legacy `createdBy` / `userId` / `editedBy` strings.
3. Enforces NOT NULL and drops obsolete columns.

There were no `prisma/migrations` in the repo at review time; for a fresh DB, `prisma migrate dev` is enough after pulling these changes.
